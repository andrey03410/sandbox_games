# Проект: Sandbox Games — личная песочница онлайн-игр

## 📌 Контекст и статус

- Это **личный pet-проект и песочница** для экспериментов с идеями многопользовательских игр.
- Пока используется **исключительно локально** (разработка на одной машине, без продакшена, без деплоя, без внешних пользователей).
- Цель в перспективе — превратить это в **мой личный сайт с онлайн-играми**: от простых (крестики-нолики, викторина) до более сложных (шахматы, пошаговые стратегии и т.д.).
- Архитектура и стек могут меняться по ходу экспериментов — это нормально, проект нужен для пробы разных подходов.

## 🎯 Идея продукта

Сайт, где:

- Любой пользователь может зайти и **создать лобби** под конкретную игру.
- В лобби **настраиваются параметры игры** (список участников, правила, таймеры, сложность и т.п. — зависит от игры).
- После старта запускается сама игра в реальном времени.
- Поддерживаются роли: **host** (создатель/ведущий лобби), **player** (участник), **spectator** (наблюдатель).
- Новые игры добавляются модульно через общий интерфейс `BaseGame`, чтобы платформа росла.

**Первая игра для MVP** — викторина в стиле «Кто хочет стать миллионером»: вопросы, таймеры, disconnect/reconnect, realtime-синхронизация.

## ⚡ Ключевые архитектурные принципы

- **Realtime через WebSocket** — мгновенное обновление состояния у всех участников комнаты.
- **Состояние игры хранится на сервере** (в памяти для MVP); клиент получает snapshot при подключении или reconnect.
- **Event-based обмен**: `join_room`, `start_game`, `next_question` / ход игрока, `answer`, `reveal_result`, `heartbeat/ping` и т.п.
- **Таймеры через timestamps**, чтобы корректно восстанавливать их после reconnect.
- **Модульность игр**: каждая игра реализует общий интерфейс `BaseGame`, совместимый с лобби и WebSocket-роутингом.
- **Роли**: host (обычно один, но потенциально несколько), player (один или несколько), spectator.

## ⚙ MVP-функциональность

**REST API**
- авторизация пользователей;
- создание комнаты / лобби;
- получение списка доступных игр и данных (например, вопросов для викторины).

**WebSocket события**
- `join_room`, `start_game`, игровые ходы / `next_question`, `answer`, `reveal_result`, `heartbeat/ping`.

**Поведение**
- таймеры и синхронизация состояния;
- обработка disconnect/reconnect со snapshot-ом;
- лобби с выбором игры и настройкой её параметров;
- разделение ролей host / player / spectator.

## 🔮 Планы на расширение

- Несколько игр одновременно на одной платформе (крестики-нолики, викторина, шахматы и т.д.).
- Дополнительные роли (админ, комментатор).
- Redis для репликации состояния и горизонтального масштабирования.
- Несколько инстансов backend-а за балансировщиком.
- Простой способ для меня (и потенциально других разработчиков в будущем) добавлять новую игру через `BaseGame`.

## 📌 Замечания

- Для MVP SQLite достаточно, но Postgres сразу удобнее — не переписывать потом.
- FastAPI полностью подходит под задачу, менять стек смысла нет.
- Главная архитектурная сложность — корректная синхронизация state при reconnect и работа с таймерами.
- Так как проект пока сугубо локальный, про безопасность/прод-grade auth/rate limiting/CORS policy можно не переусердствовать — это песочница.
